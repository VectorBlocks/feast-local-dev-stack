## Quick start

### 1. Clone and prepare

```bash
git clone https://github.com/VectorBlocks/feast-local-dev-stack.git
cd feast-local-dev-stack
```

### 2. Start services

```bash
make up
```

This builds the `feast-dev` container and starts Redis. Wait a few seconds for services to be ready.

### 3. Apply feature definitions

```bash
make apply
```

Registers the feature repository (entities, feature views) with Feast.

### 4. Materialize features

```bash
make materialize
```

Generates synthetic data, runs `feast materialize-incremental`, and writes features to Redis.

### 5. Fetch online features

```bash
make fetch
```

Runs a demo that fetches online features for sample customer IDs. You should see a dictionary of feature values printed.

### 6. (Optional) Explore with Feast Web UI

```bash
make ui
```

Starts the Feast Web UI at [http://localhost:8888](http://localhost:8888). Browse feature views, entities, and their relationships. Run after `make apply`.

### 7. (Optional) Jupyter notebook walkthrough

```bash
make notebook
```

Starts Jupyter Lab at [http://localhost:8889](http://localhost:8889). [Open walkthrough notebook](http://localhost:8889/lab/tree/notebooks/feast_walkthrough.ipynb) â€” run after `make materialize`.

## Feature definitions

This section describes the example feature structure included in this project. Use it to understand what data you work with when running `make fetch` or `make materialize`.

### Entity

| Entity | Join key | Description |
|--------|----------|-------------|
| `customer` | `customer_id` | Features are retrieved per customer (e.g., 1001, 1002, 1003) |

### Feature view: `customer_stats`

`customer_stats` is a Feature View with e-commerce customer purchase-related features.

| Feature name | Type | Description | Example values |
|--------------|------|-------------|---------------|
| `purchase_count_30d` | Int64 | Purchase count in the last 30 days | 0~15 |
| `avg_order_value_30d` | Float32 | Average order value in the last 30 days | 0~150,000 |
| `days_since_last_purchase` | Int64 | Days since last purchase | 0~30 |

### Sample data

- **3 customers**: `customer_id` 1001, 1002, 1003
- **Period**: Fixed 10 days (2026-03-01 ~ 2026-03-10), each customer has data for all 10 days
- **Generated by**: `example_repo/generate_sample_data.py` (fixed data to demonstrate Online vs Historical difference)
- **Output path**: `example_repo/data/customer_stats.parquet` (created at runtime, excluded via `.gitignore`)

### Retrieval example

```python
# Online: fetch latest features by customer_id
store.get_online_features(
    features=["customer_stats:purchase_count_30d", "customer_stats:avg_order_value_30d", "customer_stats:days_since_last_purchase"],
    entity_rows=[{"customer_id": 1001}],
)
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| `Connection refused` to Redis | Ensure `make up` completed and Redis is running. Check with `docker compose ps`. |
| `feast apply` fails | Verify you're running from a host with access to the `feast-dev` container. Use `make apply` (runs inside container). |
| No features returned from `make fetch` | Run `make materialize` first. Features must be materialized before online retrieval. |
| Port 6379 in use | Change the Redis port in `docker-compose.yml` and update `feature_store.yaml` to match. |

### Steps when changing schema

When you change the schema (Entity, Feature View, etc.), reset the existing registry and Redis data.

```bash
make down
rm -rf example_repo/data/
make up
make apply
make materialize
make fetch
```

- `example_repo/data/` contains the registry (registry.db) and Parquet files; schema changes can cause inconsistencies.
- Redis is reset by `make down -v`, but `example_repo/data/` persists on the host and must be deleted manually.
